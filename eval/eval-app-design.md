# Eval App — Design

The eval app drives benchmark runs against external adapter HTTP servers (the
sessions contract from `thesis/thesis.typ` §4.4), grades the resulting
artefacts, and serves a browse/launch UI. It is one Python package
(`benchmark/eval/eval/`) with a shared core consumed by both a CLI and a
Starlette/HTMX UI.

## Package layout

```
benchmark/eval/
  adapters.yaml                          # named adapter targets
  eval/
    core/                                # task loading, runner, scoring, storage
      tasks.py
      runner.py
      scoring.py
      storage.py
    cli.py
    ui/
      app.py
      templates/
      static/
  geo_grading/                           # existing comparison primitives
  tasks/                                 # existing task suite
  runs/                                  # eval run outputs (new home)
```

Both interfaces construct the same core objects. The CLI's "trigger a run"
path and the UI's "POST /runs" path call the same `run_one_task` machinery;
neither calls into the other.

## Adapters

`benchmark/eval/adapters.yaml` declares named adapter targets. `attributes`
is a free-form pass-through dict copied verbatim into `run.json`.

```yaml
adapters:
  claude-code-opus-basic:
    url: http://localhost:8080
    label: "Claude Code (Opus) — basic prompt"
    headers: {}                          # env-var substitution: ${VAR}
    max_concurrent_sessions: 1           # optional; CLI default
    attributes:                          # free-form, recorded verbatim
      prompt_variant: basic
      model: claude-opus-4-7
      family: claude-code
```

`--agent-url` is the ad-hoc escape hatch (no config entry needed). One CLI
call produces one run; repetition and matrix studies are external scripting.

## CLI

```
eval run <adapter|--agent-url URL> [--tasks ID...] [--max-parallel N] [--label STR]
eval score <run-id|run-dir>
eval ui [--port N]
```

- `eval run` — submit, poll, download, grade inline. One run.
- `eval score` — re-grade all tasks in a run (overwrites `score.json`).
- `eval ui` — start the Starlette app.

Run browsing happens in the UI. Reference pmtiles are (re)generated by
running each task's `visualize.py` as a standalone script during task
authoring — not via the eval CLI.

## Run lifecycle

Per-task session lifecycle:

```
POST /sessions                            → session_id
for f in inputs:  POST /sessions/{id}/files (multipart)
POST /sessions/{id}/messages {instruction}
while True:
    sleep POLL_INTERVAL
    GET /sessions/{id}                    → status, error, usage
    check cancel.flag                     → DELETE + cancelled if present
    if status in (idle, failed): break
GET /sessions/{id}/files                  → list
for f in listed: GET /sessions/{id}/files/{f} → outputs/
GET /sessions/{id}/messages               → transcript.json
write session.json
score inline (invoke grade.py via dynamic import)
visualise inline (invoke visualize.py if present)
DELETE /sessions/{id}                     # always, even on failure
```

Default concurrency is 1, resolved as: `--max-parallel` CLI flag
> `max_concurrent_sessions` from adapter config > 1. Asyncio semaphore over
the per-task coroutine, exactly like the .old/ runner.

Cancellation IPC is a `cancel.flag` file in the run directory. Ctrl-C in
the CLI, the UI's Cancel button, and cross-process cancellation all write
this file; the runner polls for it on every session-status cycle. In-flight
tasks complete their DELETE; no new tasks are launched.

## Grading

Inline after each task. The runner imports `tasks/<id>/grade.py` and calls
`grade(run_task_dir: Path) -> ScoreReport`. The result is serialised to
`score.json`. Grader exceptions are caught and written as
`{score: null, grader_error: "<traceback>"}`; the run continues.

`eval score <run-id>` walks the run tree and re-invokes every `grade.py`,
overwriting each `score.json`. Same code path as the inline scorer.

## Visualisations

Per-task `tasks/<id>/visualize.py`:

```python
def visualize(outputs_dir: Path, out_dir: Path) -> list[dict]:
    """Read outputs from outputs_dir, write pmtiles to out_dir, return layer specs."""
```

Each layer spec:

```json
{
  "name": "buildings",
  "pmtiles": "buildings.pmtiles",
  "source_layer": "buildings",
  "geometry_type": "Polygon",
  "style": { "fill-color": "#1a4480", "fill-opacity": 0.45, ... },
  "tooltip": ["name", "class"]
}
```

The list is serialised to `layers.json` in the output directory. The UI
loads `.pmtiles` over HTTP range requests with MapLibre.

- **Reference visualisations.** Pre-generated and committed under
  `tasks/<id>/reference/visualizations/`. Regenerated by invoking
  `visualize.py` directly during task authoring.
- **Agent visualisations.** Generated inline by the runner after grading,
  written to `runs/<run-id>/<task_id>/visualizations/`. Visualise-step
  failures are caught and noted in `layers.json`'s `error` field;
  they do not affect the score.

Tippecanoe is a system dependency; document it in `benchmark/eval/README.md`
and fail loudly if missing.

**Tabular outputs.** Any `expected_outputs[i]` with `format: csv`, or
geometry-less parquet, is auto-rendered as an HTML table in the UI. No
per-task config; no `visualize.py` involvement.

## On-disk layout

```
benchmark/eval/runs/run-<UTC-timestamp>/
  run.json                               # live, atomic writes
  cancel.flag                            # presence = cancel requested
  <task_id>/
    session.json                         # final GET /sessions/{id} + file list
    transcript.json                      # GET /sessions/{id}/messages
    outputs/                             # downloaded session files
    visualizations/                      # *.pmtiles + layers.json (agent)
    score.json                           # gates + subchecks
```

Run ID = UTC timestamp only (`run-20260511-1430Z`). Metadata lives inside
`run.json`, not in the directory name.

## `run.json` schema

```json
{
  "run_id": "run-20260511-1430Z",
  "schema_version": 1,
  "started_at": "2026-05-11T14:30:00Z",
  "finished_at": "2026-05-11T14:47:12Z",
  "status": "running" | "done" | "failed" | "cancelled",
  "adapter": {
    "name": "claude-code-opus-basic",     // null if --agent-url used
    "label": "Claude Code (Opus) — basic prompt",
    "url": "http://localhost:8080",
    "attributes": { "prompt_variant": "basic", "model": "...", ... }
  },
  "invocation": {
    "max_parallel": 1,
    "task_filter": ["crs-l1-*"],          // null if all tasks
    "suite_git_sha": "e9d03d6"
  },
  "tasks": {
    "crs-l1-paris-lambert93": {
      "status": "done" | "running" | "pending" | "failed" | "cancelled",
      "session_id": "s_abc123",
      "started_at": "...",
      "finished_at": "...",
      "duration_s": 142.3,
      "estimated_cost_usd": 0.42,
      "reported_model": "claude-opus-4-7",
      "reported_agent_version": "claude-code-1.0.45",
      "score": 0.875,                     // mirrored from score.json
      "error": null,
      "last_event": "downloaded paris_buildings_lambert93.geojson"
    }
  }
}
```

Atomic writes (temp + rename) after every task state transition. No
aggregation block — totals are computed on demand by consumers.

## `score.json` schema

```json
{
  "task_id": "crs-l1-paris-lambert93",
  "graded_at": "2026-05-11T14:32:11Z",
  "gates": [{ "name": "format_valid", "passed": true, "detail": "" }],
  "score": 0.875,                          // passed_subchecks / total_subchecks; null if grader crashed
  "subchecks": [
    { "name": "lambert93_envelope", "passed": true, "axis": "crs", "detail": "" }
  ],
  "grader_error": null
}
```

## UI

Pages:

1. `/` — task suite browser.
2. `/tasks/{id}` — task detail, reference map from committed pmtiles.
3. `/runs` — run list, sortable/filterable.
4. `/runs/new` — adapter dropdown (or ad-hoc URL) + task filter + max-parallel override.
5. `/runs/{id}` — run header + live HTMX-polled task table @ 1.5 s + cancel button.
6. `/runs/{id}/{task_id}` — split-view map (reference + agent overlay) + tabbed
   transcript / score / session-metadata / outputs + full-width tabular outputs below.

Live updates are HTMX polling, not SSE. The UI re-reads `run.json` from
disk on each poll cycle; no in-memory cache. CLI-launched and UI-launched
runs are observed identically. Once all tasks reach a terminal status, the
polling fragment returns without `hx-trigger`, stopping the polling loop.

Transcript rendering covers the contract's event types:

- `system` — small grey italic line.
- `text` role=user — user-tinted bubble (the instruction).
- `text` role=assistant — assistant-tinted bubble with markdown.
- `thinking` — collapsed by default, expandable, monospace.
- `tool_call` — `name(arguments)` block, JSON pretty-print, collapsed when long.
- `tool_result` — paired with its `tool_call`, monospace, red border on `is_error`,
  "truncated by adapter" annotation when flagged.

## Explicitly out of scope

- No budget gate.
- No aggregation: no `eval report`, no `results.json`, no `report.md`, no leaderboard.
- No campaign/matrix object; repetition is external scripting.
- No SSE; HTMX polling only.
- No subprocess grading; dynamic import only.
- No per-task viewer config in `task.json` / `metadata.yaml`; styling lives in `visualize.py`.
- No `eval ls`; run browsing happens in the UI.
- No `eval visualize-refs`; `visualize.py` is invoked directly during task authoring.
